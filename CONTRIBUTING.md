# Правила участі у проекті VoP СЕП НБУ

## Про проект

Цей репозиторій містить відкриту документацію та специфікації системи **Verification of Payee (VoP)** для Системи Електронних Платежів Національного банку України.

**Мета:** Забезпечити прозорість впровадження VoP та залучити банки-учасники СЕП до обговорення та розробки стандартів.

## Хто може долучитися

### Вітаємо участь від:
- ✅ Банки-учасники СЕП НБУ
- ✅ Небанківські надавачі платіжних послуг (ННПП)
- ✅ Фінтех компанії
- ✅ Розробники платіжних систем
- ✅ Академічні дослідники
- ✅ Зацікавлена громадськість

### Як долучитися:
1. Прочитайте [README.md](README.md) для ознайомлення з проектом
2. Ознайомтеся з [Quick Start Guide](training/VoP-Quick-Start.md)
3. Перегляньте відкриті [Issues](../../issues)
4. Приєднуйтеся до [Discussions](../../discussions) для обговорень

## Участь у розробці

### Створення Issue:

**Для повідомлення про проблеми:**
- Використовуйте шаблон Bug Report
- Надайте детальний опис проблеми
- Додайте кроки для відтворення

**Для пропозицій нових функцій:**
- Використовуйте шаблон Feature Request
- Опишіть use case та бізнес-потребу
- Обґрунтуйте важливість пропозиції

### Contribution Guidelines:

**⚠️ ВАЖЛИВО: НЕ включайте конфіденційну інформацію:**
- ❌ Реальні credentials, паролі, API ключі
- ❌ Приватні ключі та сертифікати
- ❌ Реальні IBAN, РНОКПП, ЄДРПОУ
- ❌ Внутрішні IP адреси та URLs
- ✅ Використовуйте тільки тестові/приклади даних

2. **Зберігання конфіденційних даних:**
   - НЕ commit'ити credentials, API keys, сертифікати
   - НЕ commit'ити реальні IBAN, імена клієнтів
   - НЕ commit'ити production конфігурацію

3. **Несанкціонований доступ:**
   - Не надавати доступ третім особам без схвалення НБУ
   - Не використовувати спільні акаунти

### ✅ РЕКОМЕНДОВАНО:

1. **Використовувати .gitignore:**
   - Завжди перевіряйте .gitignore перед commit
   - Додавайте всі конфіденційні файли до .gitignore

2. **Code Review:**
   - Всі зміни проходять через Pull Request
   - Мінімум 1 reviewer з команди НБУ

3. **Commit Messages:**
   - Українською або англійською мовою
   - Детальний опис змін
   - Посилання на issue (якщо є)

## Workflow

### 1. Створення нової функціональності

```bash
# Створити нову гілку
git checkout -b feature/new-validation-rule

# Внести зміни
# ...

# Commit
git add .
git commit -m "Додано валідацію IBAN для корпоративних рахунків"

# Push
git push origin feature/new-validation-rule

# Створити Pull Request на GitHub
```

### 2. Pull Request

**Вимоги:**
- Детальний опис змін
- Посилання на відповідні issue
- Reviewer: мінімум 1 особа з НБУ
- CI/CD checks пройдені (якщо налаштовано)

**Template PR:**
```markdown
## Опис
Додано нові правила валідації для корпоративних рахунків.

## Зміни
- [ ] Оновлено `rules/validation_rules.md`
- [ ] Додано приклади коду
- [ ] Оновлено тести

## Checklist
- [ ] Документація оновлена
- [ ] Code review пройдено
- [ ] Тести пройдені
```

### 3. Review Process

1. Reviewer перевіряє зміни
2. Коментарі та пропозиції
3. Автор вносить корективи
4. Approve → Merge to main

## Структура гілок

```
main              # Production-ready документація
├── develop       # Development гілка
├── feature/*     # Нові функції
├── bugfix/*      # Виправлення помилок
└── hotfix/*      # Термінові виправлення
```

## Версіонування

Проект використовує Semantic Versioning:
- **MAJOR.MINOR.PATCH** (приклад: 1.0.0)
- MAJOR: breaking changes
- MINOR: нова функціональність (backwards compatible)
- PATCH: bug fixes
